# common 모듈 — 가드레일

> 모든 모듈이 의존한다. 이 파일이 변경되면 전체가 재빌드된다. **최소주의가 핵심.**

## 책임

- 공통 예외 (`BusinessException`)
- 에러 코드 (`ErrorCode` enum)
- 끝.

## 의존성

**없음.** 순수 Java만. Spring 의존성도 없다.

`build.gradle.kts`에 라이브러리를 추가하기 전에 항상 자문하라:

> 이게 정말 모든 모듈에 동일한 의미로 필요한가? 한 도메인에만 필요한 게 아닌가?

## 들어와도 되는 것

- 모든 도메인이 동일한 의미로 던지는 예외
- 모든 모듈이 같은 코드값으로 참조하는 에러 enum
- (드물게) 도메인과 무관한 순수 유틸 — 단 **Rule of Three** 충족 시에만: 같은 코드가 3번 이상 반복되었고 패턴이 고착된 경우

## 절대 안 되는 것

- **도메인 정책** (예: 사용자 주소 검증 규칙) — 도메인마다 정책이 다를 수 있다
- **도메인 객체** (User, Debate 등)
- **DTO** — 도메인별로 다르며, 한쪽 변경이 다른 쪽을 깨뜨린다
- **Spring 의존성** (`@Service`, `HttpStatus` 등) — `ErrorCode`가 정수 status를 들고 있는 이유다. HTTP 매핑은 `api`의 `GlobalExceptionHandler`가 책임진다
- **JPA / DB 관련 어떤 것도**

## 새 ErrorCode 추가 패턴

```java
// ErrorCode.java
DEBATE_NOT_FOUND(404, "D001", "토론을 찾을 수 없습니다."),
ALREADY_VOTED(400, "D002", "이미 투표한 토론입니다."),
```

- HTTP status는 `int`로 보관 (spring-web 의존 회피)
- code는 도메인 prefix + 번호 (`D001` = Debate 1번)
- message는 한국어 사용자용

## "이게 common에 들어가야 할까?" 의심이 들 때

기본값은 **NO**. 도메인 모듈 안에 두고, 두 번째로 같은 패턴이 등장하면 그때도 도메인 안에 둔다. 세 번째로 등장하고 정책이 정말 동일하다는 게 확인된 후에만 `common`으로 끌어올린다.

학습 리포트에서 인용:
> 너무 일찍 추상화한 공통 모듈은 비대해진 도메인 모듈만큼이나 위험하다.
